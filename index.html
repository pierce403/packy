Packy

A local-first AI package management system designed around durable manifests, transparent installs, and user-owned state.

View Repository Agent Guide

Built For Local Control

Packy starts from the idea that AI tooling should install, cache, and explain packages where the user can inspect them.

Local-first registry state

Manifests, lockfiles, and package metadata should stay readable and portable on the local machine.

Agent-aware installs

AI assistants should be able to propose, verify, and record dependency changes without hiding the plan.

Durable package memory

Packy should remember what was installed, why it changed, and how to reproduce that state later.

Project Principles

These are the constraints future code should satisfy as Packy grows from a project page into a real package manager.

  • User-owned state Prefer files and stores that users can back up, diff, and move.
  • Explainable changes Every AI-assisted package operation should leave an auditable reason.
  • Offline-friendly flows Cached metadata and packages should keep common workflows moving without network access.
  • Small, inspectable tools Favor plain manifests and CLI behavior over opaque background automation.

Agent Etiquette

Packy follows the `recurse.bot` convention: keep `AGENTS.md` canonical, verify commands, and commit useful operating knowledge where future agents will read it.

Future agents should read the repo guidance before making broad changes, avoid inventing shipped features, and update the skill catalog only when a workflow is likely to repeat.
